# 程序员行为准则（Python/Kivy 改编版）

### 保持工作空间整洁

合理使用 Git 的能力：

1. 写新功能代码前要保证工作区干净（无未提交修改）。
2. 审查/Review 代码时，合理使用 `diff`/`log` 等工具，了解代码变更的上下文，通过变更的上下文来理解任务需求。

### 可读性优先

1. **避免重复属性访问**：不要重复地通过 `.` 获取不变的成员，应提前存储在局部变量中。

   python

   ```
   # ❌ 重复访问
   if self._ctx.business.restaurant.is_open:
       ... self._ctx.business.restaurant.name ...
   
   # ✅ 提取局部变量
   restaurant = self._ctx.business.restaurant
   if restaurant.is_open:
       ... restaurant.name ...
   ```

   

2. **变量命名要语义化**：使用 `unit_config` 而非 `uc` 或 `unit`。

3. **避免无意义的转发层**：不要创建仅做转发的静态工具类或中间层命名空间，直接在需要使用的地方引用源模块。

4. **禁止重复代码**：提取公共函数/类/模块。

5. **删除导出前必须查找所有引用方**：删除任何 `public` 方法/类（Python 中即模块级函数/类）之前，必须先用 IDE 的“查找引用”功能确认所有调用方已一并修改后再删除。直接删除会导致运行时 `AttributeError`。

### 积极拆分文件

单个代码文件不应超过 **500 行有效代码**（不含空行和注释）。大文件会稀释 AI 助手的上下文注意力，降低代码审查和修改的准确性。当文件接近上限时，应沿逻辑边界（类、功能模块、数据结构等）拆分为更小的、职责单一的模块。

**新类/算法必须创建独立文件**：新增的类或算法禁止与已有的其他类/算法混在同一文件中。新文件应以该类/算法命名（如 `bfs.py`、`priority_queue.py`）。放置路径需考虑复用范围：若可被多个项目复用，放在公共模块目录（如 `common/`）；若仅限当前项目使用，放在对应项目目录下。

**新类/算法必须配套单元测试**：新增的独立类或算法必须同步创建对应的测试文件（如 `test_priority_queue.py`）。类/算法的外部依赖（如上下文、服务、数据源）应通过**依赖注入**（如构造函数参数）而非直接导入具体实现，以便测试时 mock。

### 减少假设和约定

不要在代码中引入对外界的假设，而是通过语法或 CI 机制来消除假设和约定。使用 `Enum` 配合 `switch`（Python 中用 `match/case` 或字典映射），利用语言的穷举检查能力。

### 优先使用 Enum，禁止字符串字面量联合类型

能用 `Enum` 尽用 `Enum`。字符串字面量缺乏重构安全性，IDE 无法可靠地全局重命名，且不具备自文档化能力。

`Enum` 定义处必须包含完备的使用场景注释，说明每个枚举值的含义和用途。

**好**：

python

```
from enum import Enum

class NotifyPosition(Enum):
    """游戏内通知的显示位置"""
    CENTER = "center"      # 屏幕中央弹窗，用于重要系统通知
    TOP_RIGHT = "top_right" # 右上角提示，用于常规游戏消息
```



**差**：使用字符串常量代替 `Enum`。

### 性能优化规则

1. **优化必须彻底**：同一函数/代码块内存在多个同类低效操作时（如多处 `list.index()` 线性搜索），必须全部优化，不能遗漏。例如将 `list` 改为 `set` 时，要检查所有 `in` 操作。
2. **优化后必须删除旧代码**：新实现替代旧实现后，旧方法/旧代码必须删除，不允许保留“以防万一”的死代码。
3. **循环内避免创建对象**：循环内避免重复创建临时对象。需要索引查找时预构建 `dict`/`set`。

### 保持最小作用域

1. 变量作用域应限制在最小范围内，避免全局变量污染。
2. 限制类字段和方法的访问权限，避免不必要的 `public`（Python 中使用 `_` 前缀表示保护，`__` 表示私有）。
3. 如果一个属性可以设为只读（`@property` 无 setter），就应该设为只读。

### 保持内聚，尽量读别人系统的状态，而非让外部控制自己

例如，某个技能或能力在某种系统状态下需要停止时，应该由此能力去监听或读取系统状态，而不是反过来由系统去控制这个能力的开启。

### 保持可维护性

1. 永远不要写“魔法数字”或“魔法字符串”，要用常量或枚举来表示。
2. **修改共享接口/类型后必须全局搜索所有实现并同步更新**：在基类或协议（Protocol）中添加新方法/属性后，必须搜索所有实现类，确保全部同步更新。
3. 不要在你的局部功能代码中增加外界不知道的“映射”。应通过配置值或语法进行限制，让外界若要修改就必须修改配置或语法，从而感知到这个映射的存在。不要试图用注释说“如果修改 XX，需要同步修改此处”。
4. 必要时，在注释中解释复杂代码的设计思路，以便后续维护。
5. 不要作“约定”，尽量将“约定”转化为“强制”（例如用类型系统或断言）。
6. 如果代码中的常量、类型或数据结构来自外部 API 协议（如第三方 SDK），必须在声明处注释中标注协议文档的网址。
7. **引入缓存（lazy cache、memoization 等）时必须确保缓存有效性**：明确数据源是否真正不可变；若可变，必须实现缓存失效机制（如数据源变更时清除缓存），或证明在缓存生命周期内数据源不会变化。禁止“只加缓存不管失效”。

### 谨慎处理 UI 显示文字

任何显示给用户的文字都必须要经过翻译处理，例如通过统一的翻译配置模块来引用。在固定的地方统一引用翻译文本，可实现 CI 检查翻译是否存在，避免遗漏。

### 提交标题要求

如果提交修复了某个 issue，那么提交标题必须包含 `fix #issue_number`，例如 `fix #1234: 修复了 XXX Bug`。

### Commit 前不要破坏 CI/CD 流程

在结束你的阶段性任务、需要他人 Review 之前，你必须确保：

- 所有单元测试通过（`pytest`）
- 静态类型检查通过（如 `mypy`，若项目启用）
- 应用可正常启动（Kivy 主窗口无报错）

否则你会阻塞他人工作。

### 代码风格（Python）

1. 禁止保留代码层面的 `@deprecated` 或废弃标记，重构过程中的引用应被完整处理，而不是偷懒留下废弃标记。
2. 注释应做到“信雅达”——忠实传达设计意图、表达简洁优雅、让读者一目了然。禁止翻译式注释（即复述函数名）。需补充参数说明。

python

```
# ❌ 翻译式注释：只是复述了函数名
def get_user_name(): ...

# ✅ 信雅达注释：传达边界条件、业务语境
def get_user_name():
    """从存档中解析用户名，存档无效时返回 None"""
    ...

# ✅ 参数注释示例
def update_task(task):
    """
    更新任务信息到数据库中
    :param task: 目标任务对象，要求 task.id 已存在于数据库中
    """
    ...
```



### 文件命名与路径规范

文件名应具有自描述性，清晰传达其职责和所属模块。目录结构应遵循渐进式披露原则。

**禁止的模糊文件名**：`utils.py`、`helpers.py`、`common.py`、`config.py`、`manager.py`。

**好**：`agent_config.py`、`task_status.py`、`bot_message_format.py`
**差**：`config.py`、`constants.py`、`utils.py`

目录组织：文件应放在合适的子目录中，例如 `src/model/companion.py`，而不是根目录。如果某个模块文件数超过 3 个，应考虑创建子目录。

## Kivy / Python 应用开发规范（针对本喝水项目）

### 编译/运行验证

修改 Python 文件后，必须运行 `python -m main`（或使用 `pytest` 运行测试）确认无语法错误和运行时错误。

### Kivy Widget 生命周期

- `__init__` 用于自身属性初始化，`on_kv_post` 用于 KV 绑定后的初始化，`on_start`（App 层面）用于启动时一次性逻辑。
- `on_touch_down` / `on_touch_up` 等事件方法需谨慎重写，确保调用父类方法或在合适时机返回 `True`/`False`。
- `on_enter` / `on_leave` 用于 Screen 切换时的状态管理。

### 属性暴露与封装

- 使用 `StringProperty`、`NumericProperty`、`ObjectProperty` 等 Kivy 属性暴露给 KV 绑定，禁止直接使用 Python 普通属性进行 UI 绑定。
- 使用 `@property` 封装内部状态，避免外部直接访问私有属性。

### 引用缓存

- 在 `__init__` 或 `on_kv_post` 中缓存 `App.get_running_app()` 或常用 widget 引用（例如通过 `self.ids`），避免在频繁调用的方法（如 `on_touch_down`）中重复查找。

### 空值守卫

- 使用 `ids` 或 `ObjectProperty` 绑定时，如果可能为 `None`，在使用前必须判空。不要假设 KV 文件中的 id 总是有效。

### 资源文件管理

- 图片、音效等资源文件必须放在 `resources/` 目录，并使用相对路径引用（如 `'resources/sprites/anim1.png'`）。
- 避免硬编码绝对路径。

### 异步与持久化

- 在 Kivy 主线程中不要执行耗时 I/O（如 JSON 读写），应使用 `Clock.schedule_once` 或 `Thread` 异步执行，并在完成后通过 `Clock.schedule_once` 回到主线程更新 UI。

### 资源卸载与内存管理

- Kivy 的 `Image` 加载大量序列帧时，注意使用 `Texture` 缓存管理，避免内存泄漏。在切换形态时，主动释放不再使用的纹理（通过 `Image.texture = None` 或 `reload`）。

### 跨平台构建（Android/iOS）

- 使用 `buildozer` 或 `kivy-ios` 打包时，确保 `buildozer.spec` 或 `toolchain.py` 中的权限、依赖与项目匹配。
- 禁止在应用内申请非必要的权限（如网络、定位）。

### 适配不同屏幕尺寸

- 使用 Kivy 的 `dp`、`sp` 单位，以及 `FloatLayout`、`RelativeLayout` 等自适应布局。避免固定像素值。

## Git 操作安全规范

执行破坏性 Git 操作前必须检查工作区状态：

1. 执行 `git reset --hard`、`git checkout .`、`git restore .` 等操作前，必须先运行 `git status` 确认是否有未提交的修改。
2. 如果有未提交的修改需要保留，先执行 `git stash` 暂存，再执行破坏性操作。
3. 切换分支前确认当前分支是否有未提交的修改。

## Git 交付检查清单

任何需要提交到 Git main 分支的工作成果（代码、策划、美术）都必须经过以下检查：

1. 逐行/逐图检查自己的修改，确保 PR 中的修改都是意料之内的改动。
2. 确保项目可运行（`python -m main` 无报错），所有单元测试通过。
3. 检查 commit 信息，要求信雅达，简短有效地表达该 commit 的修改。
4. 创建 PR 时，如果有多个 commit，检查 commit 列表是否经过整理，不要在 PR 中出现 fix 当前 PR 的 commit（不要出现无意义的 Commit）。如需修复当前 PR 引入的问题，必须使用 `git rebase -i` 将修复整合到引入问题的原始提交中。
5. 如果这个 PR 跟某些 Issue 关联，在 PR 信息的 Development 中关联这些 Issue。

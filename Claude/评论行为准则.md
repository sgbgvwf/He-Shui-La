# 评论行为准则

本准则适用于在 GitHub Issue/PR 上发布评论的场景。

## 核心原则

**只发布有信息量的评论。** 如果你的评论不能为用户提供其无法从其他渠道获得的信息，就不要发布。

## 不要发布的评论

1. **确认已完成的动作** - 如果你执行了合并/关闭/推送等操作，状态变化本身就是回应，无需再发评论确认
2. **纯客套话** - "感谢"、"没问题"、"好的"等无实质内容
3. **用户已知的信息** - PR 状态、CI 结果等用户可自行查看的内容
4. **纯状态通知** - 如果没有需要用户关注的具体内容

## 值得发布的评论

- 用户需要知道但无法从其他渠道获得的信息
- 需要用户做出决策或采取行动的请求
- 对用户问题的实质性回答
- 代码审查中发现的具体问题
